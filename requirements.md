# 稼働状況分析 — 要件定義書

作成日: 2026-04-13
ステータス: ドラフト

---

## 1. 目的

IoT計測済み設備の電力データから稼働状態（停止/待機/稼働等）を分類し、
待機時間・稼働率を可視化する。将来的に予知保全・エネルギー最適化の基盤とする。

## 2. 対象範囲

- テスト段階: 1台（machine_id は Step 0 の結果から選定）
- 本番: 全 machine_id に横展開
- 設計方針: **単一実行・複数実行どちらにも対応できる構造**
  - machine_id をパラメータとして受け取る設計
  - 全台一括処理も可能にする

## 3. データソース

| 項目         | 内容                              |
|-------------|----------------------------------|
| テーブル     | `"IoT_schema".energy_raw`         |
| 粒度         | 1分                              |
| 期間         | 直近1年分                         |
| 接続方法     | PostgreSQL直結                    |

### 3.1 補助データの制約

| 補助データ       | 利用可否 | 備考                                |
|-----------------|---------|-------------------------------------|
| 平日/休日の区別  | △       | 人力でカレンダー作成すれば対応可能      |
| 日勤/夜勤の区別  | ✕       | シフトデータは保持していない           |
| 設備の定格容量   | ○       | 機械ごとの電圧容量（銘板情報）を参照可能 |

### 3.2 使用カラムと用途

| カラム            | 用途           | 実装段階       |
|------------------|---------------|---------------|
| active_power_kw  | 稼働状況の分類  | Phase 1（今回）|
| current_a        | 電流異常の検出  | Phase 2        |
| power_factor     | 力率異常の検出  | Phase 2        |

## 4. 分類仕様

### 4.1 状態の定義

- 最低2状態: **停止 / 稼働**
- 設備によって3状態以上もあり得る: 停止 / 待機 / 稼働 / 高負荷 等
- 状態数と閾値は **設備ごとにヒストグラム分析で決定** する
- 分類できない設備（山が不明瞭）は2状態（停止/稼働）にフォールバック

### 4.2 閾値の管理

- 設備ごとの閾値をテーブルまたは設定ファイルで管理
- 閾値が決まれば SQLクエリ内で CASE文による分類が可能

```
-- 閾値テーブルの想定構造
machine_thresholds:
  machine_id  | state_name | lower_kw | upper_kw
  -----------|-----------|---------|--------
  01          | stopped    | NULL     | 0.5
  01          | standby    | 0.5      | 5.0
  01          | running    | 5.0      | NULL
```

### 4.3 ヒストグラムで区切りが不明瞭な場合の対処

以下の順序で試行する。

1. **条件を分けて再描画** — 平日/休日で分割してヒストグラムを描き直す
   （日勤/夜勤のデータは無いため、時間帯による分割で代替する）
2. **KDE（カーネル密度推定）の適用** — bin幅に依存しない連続分布で谷を探す
3. **時系列の変化点検出** — 分布ではなく時系列上の急変点で状態遷移を捉える
4. **2分類へのフォールバック** — 上記で分離できない場合は
   「ほぼ0kW＝停止 / それ以外＝稼働」の単純ルールで割り切る

分布がなだらかな設備は、負荷が連続的に変動する性質であり、
無理に多分類するより2分類が正しい判断となる場合がある。

### 4.4 クラスタの意味（因果関係）が不明な場合の対処

以下の順序で推定する。

1. **時間帯との突き合わせ** — 各クラスタがいつ出現するかを分析し、
   夜間のみ出現する山＝停止、日中に出現する山＝稼働 等と推定する
2. **滞在時間の分析** — 各状態が何分〜何時間続くかを調べ、
   長時間連続＝定常状態、短時間＝過渡状態と推定する
3. **設備仕様との照合** — 分析後、機械ごとの電圧容量（銘板情報）と
   各クラスタの電力帯を突き合わせ、定格稼働・半負荷運転等を特定する
4. **実地調査による最終確認** — 上記の推定結果をヒストグラムとともに
   現場で実際の設備稼働を観察・ヒアリングして検証する

方法1・2はデータのみで実施可能。方法3・4は現場知識が必要なため、
1・2で仮説を立ててから確認する流れとする。

## 5. 成果物

### 5.1 Phase 1: 分析基盤（今回のスコープ）

1. **分析スクリプト** (Python)
   - 設備ごとに以下4種のグラフを出力（→ 5.3 グラフ出力仕様）
   - 閾値候補の自動検出（谷の位置）

2. **分類結果のDB書き込み** (SQL / Python)
   - 書き込み先テーブル（新規作成）
   - 各1分レコードに状態ラベルを付与

3. **集計テーブル** (SQL)
   - 日別・設備別の稼働率、待機時間、停止時間

4. **Grafanaダッシュボード**（要相談 → 6章参照）

### 5.2 Phase 2: 拡張（将来）

- current_a による電流異常検出
- power_factor による力率異常検出
- pg_cron によるスケジュール自動更新
- アラート通知

### 5.3 グラフ出力仕様

設備ごとに以下4種のグラフを生成する。
いずれも active_power_kw のみから生成可能（追加データ不要）。

| # | グラフ種別            | 目的                                | 備考                      |
|---|----------------------|-------------------------------------|--------------------------|
| 1 | ヒストグラム          | 電力値の分布から状態クラスタを発見     | 主要分析グラフ             |
| 2 | 持続曲線              | ヒストグラムの別視点（段差＝境界）     | 補助（同一データの別表現） |
| 3 | 時間帯プロファイル     | 各クラスタの出現時間帯を把握          | 因果推定に有効             |
| 4 | 時系列サンプル        | 実際の状態遷移パターンを直感的に確認   | 文脈把握用                |

#### グラフ共通仕様

| 項目         | 仕様                                            |
|-------------|------------------------------------------------|
| 出力形式     | PNG画像                                         |
| ラベル言語   | 英語（フォント依存を回避）                        |
| 出力構成     | 設備ごとに4グラフ1枚の概要図 + 個別詳細図          |
| 出力先       | `output/histograms/`                            |

#### 個別仕様

**1. ヒストグラム**
- bin数: 自動調整（データ量に応じて30〜100）
- 基本統計（n, min, mean, max）をグラフ内に表示

**2. 持続曲線（デュレーションカーブ）**
- 横軸: 時間の割合（%）、縦軸: active_power_kw
- 水平部分＝よくいる状態、段差＝状態の境界

**3. 時間帯プロファイル**
- 形式: 1時間単位の箱ひげ図（24本）
- 平日/休日の分離: Phase 1では区別しない
  （分類が困難な設備が出た場合に後から対応）

**4. 時系列サンプル**
- 表示期間: 直近7日間
- 横軸: 時刻、縦軸: active_power_kw
